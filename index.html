1. Introduction

1.1 Problem Statement

Current digital credential systems treat trust as binary—a credential is either valid or invalid. This approach fails to capture the nuanced reality that trust degrades over time and is affected by context, behavior, and security events. A credential issued six months ago should not carry the same weight as one issued yesterday, yet most systems have no mechanism to express this distinction.

1.2 Solution Overview

The Trust Decay Framework (TDF) introduces a continuous trust assessment model where credentials carry a confidence level that diminishes predictably over time and can be refreshed through cryptographic proofs. This enables:

  • Fine-grained trust decisions — Beyond binary valid/invalid
  • Time-aware security — Older credentials naturally require re-verification
  • Risk-responsive adaptation — Trust adjusts to detected anomalies
  • Privacy-preserving proofs — Zero-knowledge mechanisms protect sensitive data
📘 Example: Banking Wire Transfer

A bank requires Level 3 (High Trust) for wire transfers. The user holds a Level 4 credential issued 45 days ago. The system calculates the current trust level, determines it has decayed to Level 3, and grants access without additional friction. If the credential had aged 90 days, it would have decayed to Level 2 and the bank would request re-authentication.

2. Terminology

Confidence Level
A four-tier classification (Level 1 to Level 4) indicating the current trustworthiness of a credential.
Half-Life
The duration after which confidence decreases by 50% without refreshment.
Heartbeat
A cryptographic proof confirming continued holder control, used to slow or reverse trust decay.
Risk Signal
An observable event that may indicate potential security compromise.
Zero-Knowledge Proof (ZKP)
A cryptographic method that proves a statement is true without revealing the underlying data.
Pairwise Identifier
A unique identifier derived for each relationship, preventing correlation across different services.
Nullifier
A unique value that prevents replay attacks while maintaining unlinkability.
Verifiable Credential (VC)
A W3C-standard credential with cryptographic proof of issuance.

3. Confidence Levels

3.1 Level Definitions

The framework defines four confidence levels with distinct properties and proof requirements.

Level Name Half-Life Required Proofs Use Cases
Level 4 Maximum Trust 30 days NoHighRisk24h + DeviceConsistency30d + GeoStable72h Government services, critical systems
Level 3 High Trust 90 days NoHighRisk24h + DeviceConsistency7d High-value banking, corporate access
Level 2 Standard Trust 90 days NoHighRisk24h Enterprise applications, retail banking
Level 1 Basic Trust 7 days None Anonymous access, low-risk services
Why Different Half-Lives? Higher trust levels require more frequent refreshment because they grant access to more sensitive resources. Level 4 decays in 30 days, while Level 2 can last 90 days without renewal.

3.2 Trust Operations

Demotion

When trust decays beyond a threshold or risks are detected, the level decreases:

  • Level 4 → Level 3
  • Level 3 → Level 2
  • Level 2 → Level 1
  • Level 1 → Level 1 (remains at minimum)

Promotion

When valid heartbeats are provided, trust may increase by one level:

  • Level 1 → Level 2
  • Level 2 → Level 3
  • Level 3 → Level 4
  • Level 4 → Level 4

Constraint: Promotion cannot exceed the credential's original issuance level.

5. Trust Decay Calculation

5.1 Decay Formula

The current trust level is calculated using a deterministic formula:

DecayFactor = 2 ^ (-(EffectiveAge × CryptoMultiplier) / HalfLife)

Where:

  • EffectiveAge = Actual age minus time offset from valid heartbeats
  • CryptoMultiplier = 1 for quantum-resistant algorithms, 5 for quantum-vulnerable
  • HalfLife = Based on initial confidence level (30, 90, or 180 days)

5.2 Level Mapping

Decay Factor RangeDemotion StepsResult
DecayFactor > 0.750 stepsNo demotion
0.5 < DecayFactor ≤ 0.751 stepDemote one level
0.25 < DecayFactor ≤ 0.52 stepsDemote two levels
DecayFactor ≤ 0.253 stepsDemote three levels (minimum trust)
📐 Decay Example

A Level 4 credential (half-life 30 days) issued 45 days ago with no heartbeats:

EffectiveAge = 45 days
HalfLife = 30 days
DecayExponent = -(45 × 1) / 30 = -1.5
DecayFactor = 2 ^ (-1.5) = 0.3535

0.3535 > 0.25 → Demote two levels
Level 4 → Level 2

9.2 Quantum Considerations

Cryptographic profiles affect trust calculations:

  • Quantum-Resistant — CryptoMultiplier = 1 (no decay penalty)
  • Quantum-Vulnerable — CryptoMultiplier = 5 (decays 5× faster)
  • Deprecated/Revoked — Immediate demotion to Level 1

10. Privacy Architecture

10.1 Pairwise Identifiers

Instead of a global identifier, each relationship uses a unique derived identifier:

PairwiseID = HMAC(RootKey, RelationshipContext || ServiceIdentifier)

The same holder interacting with different services appears as different entities, preventing correlation without the root key.

10.2 Zero-Knowledge Proofs

  • NoHighRisk24h — Proves no high-risk alerts exist without revealing which alerts
  • DeviceConsistency — Proves device stability without revealing device fingerprint
  • GeoStable — Proves location stability without revealing actual coordinates

10.3 Nullifiers

Each proof includes a unique nullifier derived from the holder's root key and context:

Nullifier = SHA256(HolderDID || Epoch || SessionKey)

This prevents replay attacks while ensuring proofs cannot be correlated across different contexts.

12. Security Considerations

12.1 Attack Mitigations

Attack TypeMitigation
Replay AttacksNonces in heartbeats, timestamps, nullifiers in ZK proofs
Key CompromiseKey rotation protocol with grace periods and revocation
Sybil AttacksProof-of-work for node identifiers, reputation systems
Man-in-the-MiddleEnd-to-end encryption with authenticated key exchange

12.2 Key Rotation Protocol

  1. Pre-rotation — Generate new keys, create updated DID Document
  2. Rotation — Sign rotation proof, accept both old and new keys
  3. Post-rotation — Remove old key after grace period (7 days)
